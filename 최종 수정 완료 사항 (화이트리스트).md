# 추가 필요 사항 및 아키텍처 개선 계획

## 1. 애플리케이션 컨텍스트 기반 티어링 타겟팅 (YAML 화이트리스트 도입)

### 🚨 문제 정의
현재 `ScoringEngine`은 HDFS의 전체 파일(전체 FSImage)을 스캔하여 글로벌하게 티어링 룰(접근시간/크기)을 적용하고 있습니다. 이는 하둡과 같은 멀티 테넌트(Multi-tenant) 클러스터에서 각 애플리케이션(Spark, Hive 등)이 가지는 고유한 데이터 라이프사이클(Context)을 무시하는 심각한 설계 결함입니다. 인프라 데몬이 비즈니스 데이터의 의미를 모르고 권한을 남용하는 형태가 됩니다.

### 💡 해결 방안
오토티어링 대상 디렉터리를 `application.yaml`에 화이트리스트(Whitelist) 형태로 명시적으로 선언합니다. `ScoringEngine`은 지정된 경로 하위의 파일들에 대해서만 제한적으로 스코어링을 수행하도록 하여 도메인 경계를 명확히 분리합니다. **또한 화이트리스트는 오토티어링의 "권한과 책임 범위"를 명시적으로 제한하여 데몬의 권한 남용을 원천 차단하는 핵심 방어 기제 역할을 수행합니다.**

### 🛠️ 구체적 작업 내역

**1. `AppConfig.java` 및 `application.yaml` 수정**
- `Scoring` 레코드 내부에 `targetDirectories` (List<String>) 필드를 추가하여 타겟 경로를 주입받습니다.
```yaml
# application.yaml 예시 (INFRA.md E2E 테스트 배포 환경)
# INFRA.md의 test-auto-tiering.sh, test-auto-tiering-savings.sh 스크립트는 
# 테스트를 위해 임의의 디렉터리를 생성하여 동작을 검증합니다.
# 테스트 파일이 필터링되어 스크립트가 무한 대기하는 사이드이펙트를 막으려면
# 테스트용 yaml 배포 시 아래와 같이 타겟 경로를 설정해야 합니다.
scoring:
  enabled: true
  interval-seconds: 86400
  target-directories:
    - "/test/auto-tiering-e2e"  # INFRA.md test-auto-tiering.sh
    - "/test/scenario_e2e"      # INFRA.md test-auto-tiering-savings.sh
```

**2. `ScoringEngine.java` 필터링 로직 추가**
- `fetchAndParse()`로 수집된 `allFiles` 중, 경로가 화이트리스트로 시작하는 파일만 걸러내어 스코어링을 진행합니다.
```java
// ScoringEngine.java 스코어링 로직 예시
private boolean isTargetDirectory(String path) {
    // 화이트리스트가 비어있으면 안전을 위해 아무것도 처리하지 않거나 예외 처리
    return cfg.scoring().targetDirectories().stream()
            .anyMatch(path::startsWith);
}

void runCycle() throws IOException {
    List<FileMetadata> allFiles = fetcher.fetchAndParse();
    
    // 화이트리스트 경로만 필터링
    List<FileMetadata> targetFiles = allFiles.stream()
            .filter(meta -> isTargetDirectory(meta.path()))
            .collect(Collectors.toList());
            
    List<ScoredJob> jobs = score(targetFiles);
    // ... DB INSERT 로직 수행
}
```

**3. 테스트 환경 파이프라인 장애(사이드 이펙트) 방어 전략**
- **이슈:** 화이트리스트가 적용되면 기존 E2E 테스트 스크립트(`INFRA.md`의 `~/test-auto-tiering.sh`, `~/test-auto-tiering-savings.sh [A|B|C]`)가 생성하는 임시 테스트 파일과 디렉터리들이 모두 필터링되어 무시됩니다. 이로 인해 `status` 전이가 아예 시작되지 않고 파이프라인이 무한 대기하거나 실패하는 심각한 사이드 이펙트(Silent Failure)가 발생합니다.
- **해결 방안:** `~/deploy-auto-tiering.sh`를 통해 데몬을 배포할 때, E2E 검증 환경에 맞는 `application.yaml`이 함께 패키징 및 덮어써지도록 설정해야 합니다. INFRA.md의 테스트 스크립트가 사용하는 `/test/auto-tiering-e2e`, `/test/scenario_e2e`를 화이트리스트에 명시함으로써 테스트 파이프라인에 미치는 영향을 차단해야 합니다.
